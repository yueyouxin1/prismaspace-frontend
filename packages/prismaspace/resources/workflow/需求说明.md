## 1. 绝对遵循原则（必须原文保留）

无论是迁移过程中对迁移目标进行了优化或改写，都必须遵循迁移目标的核心意图、业务逻辑、UI逻辑。这是迁移任务的唯一目的，就是迁移目标的核心逻辑，但以适配我们当前项目环境的方式集成。最终期望得到与迁移目标在代码逻辑和UI交互几乎一致的落地。

**依赖完整性原则（必须新增）：** 迁移不仅包含“表层代码/页面”，还必须覆盖目标实现所依赖的必要上下游与支撑能力（包括但不限于：组件/工具库、样式与主题变量、图片/字体/动画资产、数据模型与接口契约、权限与路由约束、国际化/配置项、运行时约束）。若为保证等价行为必须整体迁移模块/包，则应执行“整包迁移”，不得只做表层拼装。

在本次 `workflow` 迁移中，以上原则必须进一步收紧为以下硬约束：

1. **源码先行原则**：不得凭截图、印象、经验或“类似产品常识”推断行为。所有页面结构、组件职责、交互时序、状态切换、异常处理、字段形态，必须先回溯 `reference_example/coze-studio/frontend/packages/workflow` 对应源码，再进行迁移实现。
2. **全量复刻原则**：本次不是“做一个类似 workflow 编辑器”，而是对 `coze/workflow` 用户可见工作台、节点体系及其完整依赖链进行整包级复刻式迁移。任何用户能感知到的缺失，都视为迁移未完成。
3. **禁止主观猜测原则**：只要源码证据不足，就必须继续阅读上游实现、调用链、依赖目录和相关组件；在证据不足前，不允许主观补完，不允许“先按自己的理解做”。
4. **技术栈服从原则**：复刻目标是 Coze Workflow 的 UI 和交互逻辑，不是复制其 React/Rush 工程。实现必须服从当前项目的 Vue 包架构、依赖边界、导出规范和资源工作台体系。
5. **后端契约服从原则**：前端工作台的行为、数据流、运行链路、调试链路，必须遵循当前项目 `resources/workflow` 与 `src/app/api/v1/workflow` 的真实后端逻辑；不允许为了“看起来像 Coze”而伪造与本项目不一致的接口语义。
6. **缺失即未完成原则**：只要仍存在缺失的节点、缺失的配置面板、缺失的变量体系、缺失的运行历史/回放/调试能力、缺失的关键 UI 状态，均不得宣称本次迁移完成。

## 2. 迁移目标与边界

### 2.1 迁移目标

- 目标模块/页面：`Coze Workflow 前端整包家族`
- 本次必须纳入阅读与映射的上游目录：
  - `base`
  - `components`
  - `fabric-canvas`
  - `feature-encapsulate`
  - `history`
  - `nodes`
  - `render`
  - `setters`
  - `test-run`
  - `test-run-next`
  - `variable`
  - `adapter/*`
  - `sdk`
  - `playground`（用于理解页面装配与交互入口，不要求原样复制工程）

- 目标核心能力：
  1. `完整工作台`：顶部 header/action bar、左侧节点/资源区、中部画布、右侧节点配置区、底部或侧边问题/调试/运行/历史区。
  2. `节点系统`：节点注册、节点卡片、节点分组、端口、连线、拖拽创建、节点态、节点删除/保护规则、节点配置、节点调试。
  3. `Setter / 配置体系`：与 Coze 对齐的字段布局、折叠分组、字段类型、变量插入、表达式输入、资源选择、参数 Schema 配置、特殊字段交互。
  4. `变量体系`：工作流输入、节点输出、变量浏览、变量引用、值引用拣选、变量联动和可视化预览。
  5. `运行链路`：整图校验、同步执行、异步执行、单节点调试、运行状态查看、运行详情、事件流、回放、取消运行。
  6. `资源工作台集成`：工作区、资源实例、保存、发布、版本上下文、国际化、错误提示、空态、骨架屏、加载态。
  7. `视觉与体验`：尽可能像素级复刻 Coze Workflow 的布局密度、面板比例、间距、状态色、hover/selected/focus/disabled 态、信息层级和交互反馈节奏。

- 目标关键交互：
  1. `从左侧节点区拖拽节点到画布 -> 连线 -> 选中节点 -> 打开右侧表单 -> 配置节点 -> 保存`
  2. `在右侧配置面板插入变量/表达式/参数 Schema -> 观察表单联动、校验反馈、字段显隐和节点输入输出变化`
  3. `整图运行 / 节点调试 -> 查看运行结果、事件流、历史记录、回放与取消`
  4. `切换工作流资源实例 -> 拉取 graph、节点定义、最新版本与运行记录`
  5. `对照 Coze 页面逐区域检查 UI 与交互是否一致，不接受“差不多”`

### 2.2 当前问题基线

当前 `@prismaspace/workflow` 虽然已经有 `WorkflowHeaderBar`、`WorkflowCanvas`、`WorkflowNodeSidePanel`、`WorkflowProblemPanel`、`WorkflowTestRunPanel`、`WorkflowVariableExplorer` 等结构，但整体仍不足以满足本次迁移目标，主要问题包括：

1. 当前实现仍接近 scaffold / 第一轮承接形态，距离 Coze Workflow 的完整工作台与完整依赖链仍有明显差距。
2. 节点体系、右侧 setter 体系、变量体验、运行历史/回放/调试、细节状态和视觉完成度仍不够彻底。
3. 若仅在现有基础上做局部修补，容易继续停留在“有功能但不完整、不精细、不对等”的状态。

因此，本次文档明确要求：后续开发必须以 Coze 源码为基准逐模块补齐，不得再以主观想象或局部近似替代整包复刻。

### 2.3 迁移边界

- 本次 In Scope：
  1. `全量整包复刻 UI`：至少覆盖工作台、节点、完整依赖组件、右侧配置体系、变量体系、问题区、运行区、历史与回放区。
  2. `源码对照迁移`：必须先阅读 Coze Workflow 源码，再做模块映射和逐项落地，不接受仅按产品理解重做。
  3. `按当前项目技术栈实现`：使用当前项目的 Vue 3 + TypeScript + 公开包架构 + 既有 UI/数据流方案完成落地。
  4. `按当前后端逻辑对齐`：前端数据流、运行链路、调试链路必须对齐 `src/app/api/v1/workflow` 与通用 resource/publish 逻辑。
  5. `优先复用 editor/generator`：表达式、代码、Schema、复杂表单能力优先承接到 `@prismaspace/editor` 与 `@prismaspace/generator`。
  6. `保留后端表单契约升级窗口`：仅在前端表单生成器契约稳定后，才允许对齐后端 `workflow/nodes` 的注册节点表单定义。

- 本次 Out of Scope：
  1. `原样复制 Coze 的 React/Rush/Storybook/Vitest 工程基础设施`。
  2. `伪造 Coze 私有后端能力`，或脱离当前 `resources/workflow` 业务语义单独模拟一套数据模型。
  3. `在 generator 契约未稳定前` 贸然大改后端节点表单契约。
  4. `借迁移之名重做产品设计`、重写信息架构、改动关键交互顺序或视觉语言。

## 4. 迁移策略（源码对照的一比一复刻式落地）

### 4.1 总体策略

本次迁移不再接受“做一个功能大体相似的 workflow 工作台”。本次策略是：

1. **先读源码，再写代码**：对 Coze Workflow 各目录建立“源目录 -> 当前项目承接目录 -> 责任人/状态”的映射清单。
2. **先补依赖链，再补表层页面**：不能只做 header、canvas、panel 外壳，必须同步补齐 nodes、setters、variable、history、test-run、render、adapter 等支撑链路。
3. **先做结构对齐，再做视觉打磨**：先保证工作台区域划分、节点和侧边面板职责、调试/历史链路齐全，再逐步逼近 Coze 的细节和像素感。
4. **先做前端适配层，再动后端契约**：节点表单优先通过前端 adapter 对齐 `generator`；只有当前端契约稳定后，才允许推动后端 `workflow/nodes` 注册定义升级。
5. **每一处差异都要有源码或后端约束依据**：没有依据的差异不能落地。

### 4.2 必做的源码阅读与模块映射

| Coze 源目录 | 必须弄清的内容 | Prismaspace 承接方向 |
|---|---|---|
| `base` | 基础类型、工作流实体、上下文、校验、共用 service/store/utils | `src/types`、`src/services`、`src/utils`、`src/state` |
| `components` | 工作台通用组件、面板壳层、弹窗、空态、状态组件 | `src/components` |
| `fabric-canvas` / `render` | 画布层、节点渲染、边、交互反馈、画布工具行为 | `src/canvas`、`src/render` |
| `nodes` | 节点注册、节点 UI、节点元信息、节点输入输出、节点类别 | `src/nodes` |
| `setters` | 右侧配置面板字段、setter 组件、字段布局、复杂表单交互 | `src/panels`、`src/fields`、`src/adapters/forms` |
| `variable` | 变量树、变量来源、变量引用、表达式和值引用选择 | `src/variable` |
| `history` | 历史记录列表、详情、时间线、状态和筛选 | `src/runs`、`src/history` |
| `test-run` / `test-run-next` | 运行调试、测试表单、trace、event replay、结果查看 | `src/debug`、`src/trace`、`src/runs` |
| `feature-encapsulate` | 封装型功能模块和跨区域组合逻辑 | `src/features` |
| `adapter/*` | 资源、节点、代码编辑器、基础接口等外部接入层 | `src/adapters` |
| `sdk` | 上游调用方式与契约形态参考 | `src/services`、`@prismaspace/sdk` 对齐层 |
| `playground` | 页面装配、入口行为、联调方式、依赖串联 | 当前项目 Demo / 消费入口 |

要求：

1. 上表所有目录必须至少完成一次阅读和映射记录。
2. 任何目录若判断“不需要迁移”，也必须给出理由，并说明为何不会造成用户可见缺失。
3. 只要用户在 Coze Workflow 中能看到或能操作到的能力，都不能因为目录多、依赖深而跳过。

### 4.3 复刻标准

- **UI 复刻**：
  - 工作台整体布局、面板层级、主要间距、信息密度、视觉重心与 Coze 对齐。
  - 节点卡片、端口、连线、hover/selected/disabled/error/running 等状态视觉接近 Coze。
  - 右侧配置面板的标题层级、分组、字段节奏、描述文案、空态与帮助提示接近 Coze。
  - 运行区、问题区、历史区、trace 区的结构与交互反馈接近 Coze。

- **交互复刻**：
  - 节点拖拽、投放、连线、删除、选中、取消选中、节点调试、测试运行等流程时序与 Coze 接近。
  - 表单字段显隐、禁用、默认值、校验反馈、变量插入、Schema 编辑的行为与 Coze 接近。
  - 运行链路中的 loading、success、error、streaming、cancel、replay 等关键状态与 Coze 接近。

- **逻辑复刻**：
  - 用户可见的业务规则、边界处理、异常反馈和结果语义与 Coze 对齐。
  - 若因后端差异无法完全一致，必须保持用户感知层的结构和交互语义一致，并记录差异。

- **证据复刻**：
  - 每个已落地模块都要能指出其参考的 Coze 源码位置。
  - 任何未能给出上游源码依据的行为，不得标记为“已完成复刻”。

### 4.4 editor / generator 承接策略

- `@prismaspace/editor`：
  1. 代码、表达式、Prompt、JSON、参数 Schema 等编辑能力优先复用 editor 现有组件。
  2. 若 Coze 存在复杂 setter/editor，先判断能否由 `MonacoEditor`、`CodeMirrorEditor`、`ParamSchema*Editor`、`ValueRefPickerDialog` 组合承接。
  3. 只有 editor 现有能力确实无法承接时，才允许在 `workflow` 包内新增专用 UI 适配。

- `@prismaspace/generator`：
  1. 节点右侧配置面板优先通过 schema adapter 转换为 generator 能消费的表单描述。
  2. 对于 `resource_selector`、`model_selector`、`parameter_schema`、`workflow_json` 等复杂字段，可使用自定义 field renderer 承接。
  3. generator 契约未稳定前，禁止反向驱动后端频繁变更节点表单定义。
  4. 只有当 generator 契约已稳定并经过多节点、多字段形态验证后，才允许推进后端 `workflow/nodes` 注册定义升级。

### 4.5 分阶段执行建议

1. `阶段一：源码盘点`
   - 建立 Coze 源目录映射清单
   - 标记当前 `@prismaspace/workflow` 缺口
2. `阶段二：工作台与节点系统`
   - 完成 header / 左侧 palette / canvas / node renderer / 右侧 panel 主链路对齐
3. `阶段三：变量、setter、运行链路`
   - 补齐 variable / test-run / history / trace / replay / cancel
4. `阶段四：像素与交互打磨`
   - 对齐样式细节、过渡反馈、空态、异常提示、加载态
5. `阶段五：契约治理`
   - 当前端适配层稳定后，再评估后端 `workflow/nodes` 契约对齐

## 5. 允许做的事（Allowed）

在不破坏“核心意图、业务逻辑、UI逻辑”的前提下，允许：

1. **工程适配**：按当前项目目录结构、状态管理、路由、构建方式改写。
2. **技术替换**：用当前项目统一技术方案承接 Coze 目标能力，例如 `@vue-flow/core`、`@prismaspace/ui-shadcn`、`@prismaspace/ui-ai-elements`、`@prismaspace/editor`、`@prismaspace/generator`。
3. **适配层建设**：为节点表单、变量引用、资源选择、运行链路建立前端 adapter，而不是强行复制 React 实现。
4. **可修复问题修正**：若发现上游实现存在明确逻辑问题，可做等价修复，但必须记录依据。
5. **无副作用增强**：可做性能、健壮性、可维护性增强，前提是不改变用户可见行为。
6. **后端契约升级**：仅在前端表单生成器契约稳定后，允许对齐后端 `workflow/nodes` 注册节点表单定义。

> 所有优化、增强、适配都必须满足：默认行为不变、对外接口不破坏、用户交互预期不改变、且能回溯到上游源码或本项目后端约束。

## 6. 不允许做的事（Not Allowed）

以下行为一律禁止：

1. 未阅读 Coze 对应源码，就直接按经验或截图实现页面与逻辑。
2. 以“优化”为名改变迁移目标的核心业务规则、UI 信息架构或交互顺序。
3. 只迁移主工作台外壳，遗漏节点系统、setter、变量体系、运行历史、trace、调试等依赖链。
4. 将 Coze 中存在的关键 UI 状态、关键交互反馈、关键异常处理路径省略掉。
5. 仅因个人偏好改写布局、文案层级、面板比例、交互反馈节奏，导致与 Coze 差距明显。
6. 在没有证据的情况下自创节点字段、自创表单联动、自创运行反馈语义。
7. 引入与当前项目规范冲突的技术方案或依赖，尤其是直接把 React/Rush 方案搬入当前仓库。
8. 脱离当前 `resources/workflow` 与 `api/v1/workflow` 逻辑，单独模拟一套接口或状态体系。
9. 在 generator 契约尚未稳定时，提前推动后端节点表单契约大幅改动。
10. 将“功能部分可用但缺失较多”标记为迁移完成。
