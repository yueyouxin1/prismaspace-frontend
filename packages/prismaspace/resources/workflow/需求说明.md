# Workflow 迁移需求说明

> 用途：用于在当前项目中发起“从外部/历史实现迁移能力”的需求，确保迁移结果在**核心意图、业务逻辑、UI 交互**上与目标保持高度一致，同时完成项目环境适配。  
> 当前迁移目标：`reference_example/coze-studio/frontend/packages/workflow`  
> 当前落地目标：`prismaspace-frontend/packages/prismaspace/resources/workflow`

## 1. 绝对遵循原则（必须原文保留）

无论是迁移过程中对迁移目标进行了优化或改写，都必须遵循迁移目标的核心意图、业务逻辑、UI逻辑。这是迁移任务的唯一目的，就是迁移目标的核心逻辑，但以适配我们当前项目环境的方式集成。最终期望得到与迁移目标在代码逻辑和UI交互几乎一致的落地。

**依赖完整性原则（必须新增）：** 迁移不仅包含“表层代码/页面”，还必须覆盖目标实现所依赖的必要上下游与支撑能力（包括但不限于：组件/工具库、样式与主题变量、图片/字体/动画资产、数据模型与接口契约、权限与路由约束、国际化/配置项、运行时约束）。若为保证等价行为必须整体迁移模块/包，则应执行“整包迁移”，不得只做表层拼装。

## 2. 迁移目标与边界

### 2.1 迁移目标

- 目标模块/页面：`Coze Workflow 前端整包（workbench / nodes / components / render / variable / history / test-run / adapters 等）`
- 目标核心能力：
  1. `工作流工作台`：顶部操作区、左侧节点/资源区、中心画布、右侧配置面板、底部或侧边调试/运行区
  2. `节点系统`：节点注册、节点卡片、端口、节点分类、拖拽创建、节点配置、节点调试
  3. `依赖组件`：变量选择器、表达式编辑器、资源引用组件、运行历史、事件回放、测试运行、状态提示与空态
  4. `执行链路`：校验、同步执行、异步执行、节点级 debug、运行记录查看、事件流回放、取消执行
  5. `资源工作台集成`：遵循当前项目资源实例、发布、工作区上下文与国际化体系
- 目标关键交互：
  1. `节点拖拽入画布 -> 连线 -> 选中节点 -> 右侧表单配置 -> 保存/发布`
  2. `变量引用/表达式输入 -> 表单联动 -> 节点输入输出预览`
  3. `整图运行 / 单节点调试 -> 查看运行结果、事件流、历史记录`
  4. `切换工作流资源实例 -> 读取资源详情、节点定义、最近发布版本`
  5. `对齐 Coze Workflow 的整体工作台观感与信息结构，但实际行为遵循 Prismaspace 资源/workflow 后端`
- 当前现状：`@prismaspace/workflow` 当前仅有 workbench scaffold，只有资源信息、validate、execute、publish 的占位式面板，尚未形成完整工作流编辑工作台`

### 2.2 迁移边界

- 本次 In Scope：
  1. `全量整包复刻 UI`：至少覆盖工作台、节点、画布、配置面板、变量体系、调试/运行/历史等完整依赖组件
  2. `按当前项目技术栈重构实现`：使用 Vue 3 + TypeScript + Vite + 当前仓库 UI/状态/查询方案，不引入 React/Rush 运行方式
  3. `按当前后端逻辑对齐`：前端数据流、执行链路、节点定义、运行历史必须对齐 `src/app/api/v1/workflow` 与当前资源实例逻辑
  4. `迁移目标依赖链一并落地`：对于 Coze workflow 下的 `base`、`components`、`render`、`nodes`、`setters`、`variable`、`history`、`test-run`、`adapter/*` 等必要能力，必须整体映射并落地
  5. `节点表单适配层建设`：优先在前端建立 `workflow node definition -> generator schema / custom editor` 适配层，为后续表单契约对齐提供稳定接口
  6. `保留对后端节点表单契约升级的窗口`：在确认前端表单生成器契约规范稳定后，允许对齐后端 `workflow/nodes` 注册节点表单定义
- 本次 Out of Scope：
  1. `逐文件、逐技术栈复制 Coze 工程基建`，包括 Rush/Storybook/测试脚手架原样搬运
  2. `脱离当前后端资源/workflow 语义` 去模拟 Coze 私有接口或 Coze 专属业务规则
  3. `在 generator 契约未稳定前` 提前改写后端节点表单契约并造成双端反复震荡
  4. `借迁移额外发明新的工作流产品能力`，例如新增参考目标中不存在的编排范式或重做领域模型

## 3. 项目环境约束（强制）

迁移任务只强制引用以下开发规范文档：

- `prismaspace-frontend/doc/Vue通用架构与开发规范指南.md`
  - 约束目录、导入边界、Demo 交付与工程实现方式

其他文档如 UI、runtime、边界说明，可按需要作为参考资料补充，不作为本模板的强制前置清单。

本次迁移额外必须遵守以下现实约束：

1. 前端实现必须落在 `@prismaspace/workflow` 包内，并通过包导出对外提供能力。
2. 当前仓库技术栈以 `Vue 3 + TypeScript + @tanstack/vue-query + shadcn-vue/reka + vue-flow` 为主，迁移实现必须服从这一栈。
3. 当前 `resources/workflow` 是资源工作台子包，不得绕开现有 `resource` / `workspace instance` / `publish` 逻辑另起一套状态体系。
4. 与编辑器相关的 DSL、表达式、Schema、代码输入能力，应优先复用 `@prismaspace/editor`。
5. 与节点右侧表单面板相关的配置驱动表单能力，应优先复用 `@prismaspace/generator/form-generator`，只有在契约尚未稳定或控件能力不足时才允许局部自定义。

当前后端已可对齐的 workflow API 范围（前端需围绕这些能力设计）：

- `GET /api/v1/workflow/nodes`：获取节点定义
- `POST /api/v1/workflow/{uuid}/validate`：校验工作流
- `POST /api/v1/workflow/{uuid}/execute`：同步执行
- `POST /api/v1/workflow/{uuid}/async`：异步执行
- `POST /api/v1/workflow/{uuid}/nodes/{node_id}/debug`：节点级调试执行
- `GET /api/v1/workflow/{uuid}/runs`：运行记录列表
- `GET /api/v1/workflow/runs/{run_id}`：运行详情
- `GET /api/v1/workflow/runs/{run_id}/events`：运行事件列表
- `GET /api/v1/workflow/runs/{run_id}/replay`：事件回放
- `POST /api/v1/workflow/runs/{run_id}/cancel`：取消运行
- `POST /api/v1/workflow/{uuid}/sse`、`WS /api/v1/workflow/ws`：流式事件能力
- 工作流资源基础 CRUD / publish：继续复用通用资源路由与 `resource` 能力

冲突处理规则：

1. 先保证迁移目标的核心意图、业务逻辑、UI逻辑不变。
2. 在不改变核心逻辑前提下，按当前项目规范重构实现方式。
3. 若出现不可兼容冲突，需在需求/PR 中记录差异与原因，并给出等价方案。
4. 若 Coze 前端存在当前后端不支持的能力，优先保留 UI 结构与交互语义，再通过禁用、降级、占位或延后接入的方式处理，不得伪造后端行为。

## 4. 迁移策略（非“完全复刻”，是“等价落地”）

### 4.1 总体策略

- 不要求逐行、逐文件、逐技术栈“完全复刻”。
- 要求核心逻辑、用户可见行为、关键交互路径与目标几乎一致。
- 以匹配当前项目环境的方式进行组件替换、架构适配、工程化集成。
- 依赖链完整迁移：确保目标实现所依赖的关键能力与资源一并迁移或等价落地，必要时执行整包迁移。
- 迁移方式采用“`Coze package family -> Prismaspace domain package`”结构映射，而不是只移植单一页面：

| Coze 源目录 | Prismaspace 承接方向 |
|---|---|
| `base` / `sdk` / `adapter/*` | `resources/workflow/src/services`、`src/adapters`、`src/types` |
| `components` / `feature-encapsulate` | `resources/workflow/src/components` |
| `fabric-canvas` / `render` | `resources/workflow/src/canvas`、`src/render` |
| `nodes` / `setters` | `resources/workflow/src/nodes`、`src/panels` |
| `variable` | `resources/workflow/src/variable` |
| `history` / `test-run` / `test-run-next` | `resources/workflow/src/runs`、`src/debug`、`src/trace` |
| `playground` | 作为联调/演示入口落到当前项目实际消费端，不直接复制 playground 工程 |

- `@prismaspace/editor` 使用策略：
  1. 表达式、代码、Prompt、JSON/Schema 编辑统一优先走 editor 公共能力
  2. 若 Coze 存在高度定制的 setter/editor，先评估是否能由 `CodeMirrorEditor`、`MonacoEditor`、`ParamSchema*Editor` 组合承接
- `@prismaspace/generator` 使用策略：
  1. 节点配置面板优先落到 generator 的 schema 驱动表单
  2. 对于 generator 尚未覆盖的高级交互，允许局部自定义字段或临时自定义面板
  3. 只有当前端表单生成器契约被验证稳定后，才允许推动后端 `workflow/nodes` 注册定义对齐新的表单契约

### 4.2 等价标准

- **业务等价**：输入、处理、输出及异常分支一致。
- **交互等价**：操作流程、反馈时机、状态切换一致。
- **UI 逻辑等价**：结构层次、可见条件、关键状态表现一致。
- **依赖完整性等价**：不是只看到主页面，而是工作台可运行所需的节点、面板、变量、调试、历史等依赖链均已落地。
- **契约治理等价**：节点表单契约的演进节奏可控，不允许前端先失稳、后端再被迫跟随修补。

## 5. 允许做的事（Allowed）

在不破坏“核心意图、业务逻辑、UI逻辑”的前提下，允许：

1. **工程适配**：按当前项目目录结构、状态管理、路由、构建方式改写。
2. **技术替换**：将目标实现替换为本项目统一技术方案（如 `vue-flow`、`shadcn-vue`、`@tanstack/vue-query`、`@prismaspace/editor`、`@prismaspace/generator`）。
3. **可修复问题修正**：若发现目标存在明确逻辑问题，可做等价修复。
4. **无副作用增强**：可做性能、可维护性、可读性优化，且不改变外部行为。
5. **健壮性补强**：补齐空值保护、边界条件、并发态保护、流式连接恢复等安全性增强。
6. **可观测性增强**：补充必要日志、错误提示、调试信息（不影响用户主流程）。
7. **契约条件升级**：在前端表单生成器契约规范稳定后，允许同步调整后端 `workflow/nodes` 的注册节点表单定义，但必须有双端对齐方案与回归验证。

> 所有优化/增强都必须满足：默认行为不变、对外接口不破坏、用户交互预期不改变。

---

## 6. 不允许做的事（Not Allowed）

以下行为一律禁止：

1. 以“优化”为名改变迁移目标的核心业务规则。
2. 擅自删减关键交互流程、状态反馈或异常处理路径。
3. 改变用户可见的关键 UI 逻辑（显示条件、操作顺序、结果语义）。
4. 未评审即新增需求范围外能力，导致“借迁移做重构”。
5. 仅因个人偏好替换成熟逻辑，造成行为偏差。
6. 引入与当前项目规范冲突的技术方案或依赖。
7. 只迁移工作台表层，缺失节点系统、变量、调试、运行历史、配置面板等关键依赖链。
8. 脱离当前 `resources/workflow` 与后端 `api/v1/workflow` 逻辑，单独模拟 Coze 私有接口行为。
9. 在 generator 契约尚未稳定时，提前推动后端节点表单契约大幅变更。
